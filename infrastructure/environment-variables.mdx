Suron Vault uses different environment variables for the SDK (server runtime), CLI (dev machine), and Worker (infrastructure). This page documents all variables and where they're used.

## Overview

| Variable | Used By | Required | Storage |
|----------|---------|----------|----------|
| `VAULT_WORKER_URL` | SDK, CLI script | Yes | Server `.env`, CLI auth file |
| `VAULT_ACCESS_TOKEN` | SDK | Yes | Server `.env` |
| `VAULT_APP_ID` | SDK | Yes | Server `.env` |
| `VAULT_CLI_TOKEN` | CLI (internal) | Auto | CLI auth file |
| `VAULT_PERMIT` | SDK (optional) | No | Server `.env` (optional) |
| `BOT_TOKEN` | Worker | Yes | Worker secret |
| `ADMIN_CHAT_ID` | Worker | Yes | Worker config |
| `ADMIN_USERNAME` | Worker | Yes | Worker secret |
| `ADMIN_PASSWORD` | Worker | Yes | Worker secret |
| `CONVEX_URL` | Worker | Yes | Worker secret |
| `CONVEX_DEPLOY_KEY` | Worker | Yes | Worker secret |

## SDK Variables (Server Runtime)

These variables must be set on your **production server** where the app runs.

### VAULT_WORKER_URL

**Required**: Yes  
**Format**: `https://suronai-vault.your-account.workers.dev`

URL of your deployed Cloudflare Worker.

```env
VAULT_WORKER_URL=https://suronai-vault.your-account.workers.dev
```

<Info>
  After deploying the Worker, `wrangler deploy` outputs this URL. Copy it exactly.
</Info>

### VAULT_ACCESS_TOKEN

**Required**: Yes  
**Format**: 64-character hex string  
**Generate**: `openssl rand -hex 32`

Authentication token for SDK routes. This token is set as a Worker secret and must match on the server.

```env
VAULT_ACCESS_TOKEN=abcdef0123456789abcdef0123456789abcdef0123456789abcdef0123456789
```

<Warning>
  Keep this secret. Never commit to git. This token allows servers to request vault access and fetch secrets.
</Warning>

**How it works**:
- SDK sends this in `x-vault-token` header with every request
- Worker validates it matches `env.VAULT_ACCESS_TOKEN` (Worker secret)
- Only grants access to SDK routes (not admin routes)

### VAULT_APP_ID

**Required**: Yes  
**Format**: String identifier (alphanumeric, hyphens, underscores)

Unique identifier for your app. Must match the ID used in `suron init`.

```env
VAULT_APP_ID=my-api
```

**Examples**:
- `my-api`
- `production-web`
- `background-worker`
- `staging-api`

### VAULT_PERMIT (Optional)

**Required**: No (advanced use case)  
**Format**: Hex string (generated by Worker)

Permit token for persisting trusted restart capability across process exits.

```env
VAULT_PERMIT=def456...
```

**When to use**:
- By default, permit tokens exist only in RAM and are lost when the process exits
- Set this variable to persist the permit token across container restarts
- Example: [06-trusted-restart](/examples/trusted-restart)

**How to get it**:
```javascript
import { vault } from '@suronai/sdk'

await vault.load()

const permitToken = vault.getPermitToken()
console.log('VAULT_PERMIT=' + permitToken)
```

**Security note**: Permit tokens grant 24-hour vault access without approval. Store securely if persisting.

## CLI Variables (Dev Machine)

The CLI stores credentials in `~/.suron/auth.json` after login. You don't set these manually.

### Auth File Location

```
~/.suron/auth.json
```

Example content:
```json
{
  "workerUrl": "https://suronai-vault.your-account.workers.dev",
  "token": "abcdef0123456789abcdef0123456789abcdef0123456789abcdef0123456789"
}
```

### How CLI Auth Works

<Steps>
  <Step title="Run suron login">
    ```bash
    suron login
    ```
    
    Prompts for:
    - Worker URL
    - Username
    - Password
  </Step>
  
  <Step title="CLI calls /admin/login">
    ```javascript
    POST /admin/login
    {
      "username": "admin",
      "password": "your_secure_password"
    }
    
    Response:
    {
      "token": "<VAULT_CLI_TOKEN value>"
    }
    ```
  </Step>
  
  <Step title="Token stored locally">
    CLI writes `~/.suron/auth.json` with Worker URL and token.
  </Step>
  
  <Step title="Subsequent commands use token">
    All CLI commands send token in `x-vault-token` header.
  </Step>
</Steps>

### VAULT_CLI_TOKEN (Internal)

You never see this directly - it's returned by `/admin/login` and stored in the auth file.

**Security**: This token grants access to admin routes (`/admin/*`). It never lives on production servers.

## Worker Variables (Cloudflare Infrastructure)

These are configured in the Worker deployment.

### Worker Secrets (via wrangler secret)

Secrets are stored encrypted by Cloudflare and never appear in code or logs.

#### BOT_TOKEN

**Required**: Yes  
**Format**: `123456789:ABCdefGHIjklMNOpqrsTUVwxyz`  
**Source**: [@BotFather](https://t.me/botfather)

Telegram bot token for sending notifications and receiving webhook updates.

```bash
wrangler secret put BOT_TOKEN
# Paste token from BotFather
```

#### VAULT_ACCESS_TOKEN

**Required**: Yes  
**Format**: 64-character hex string  
**Generate**: `openssl rand -hex 32`

Server SDK authentication token. Must match the value set on production servers.

```bash
wrangler secret put VAULT_ACCESS_TOKEN
# Paste generated token
```

#### VAULT_CLI_TOKEN

**Required**: Yes  
**Format**: 64-character hex string  
**Generate**: `openssl rand -hex 32`

CLI authentication token. Returned by `/admin/login` after username/password verification.

```bash
wrangler secret put VAULT_CLI_TOKEN
# Paste generated token
```

<Info>
  Use a different token than `VAULT_ACCESS_TOKEN`. This ensures compromised servers cannot access admin routes.
</Info>

#### ADMIN_USERNAME

**Required**: Yes  
**Format**: String

Username for CLI login.

```bash
wrangler secret put ADMIN_USERNAME
# Type your chosen username
```

#### ADMIN_PASSWORD

**Required**: Yes  
**Format**: String

Password for CLI login. Choose a strong password.

```bash
wrangler secret put ADMIN_PASSWORD
# Type your secure password
```

#### CONVEX_URL

**Required**: Yes  
**Format**: `https://happy-animal-123.convex.cloud`

Your Convex deployment URL.

```bash
wrangler secret put CONVEX_URL
# Paste URL from Convex dashboard
```

#### CONVEX_DEPLOY_KEY

**Required**: Yes  
**Format**: `prod:...` (starts with `prod:`)

Deploy key from Convex dashboard (Settings > Deploy Keys).

```bash
wrangler secret put CONVEX_DEPLOY_KEY
# Paste deploy key
```

<Warning>
  Deploy keys grant full read/write access to your Convex backend. Never commit to git.
</Warning>

### Worker Config Variables (wrangler.toml)

These are non-sensitive and stored in `wrangler.toml`.

#### ADMIN_CHAT_ID

**Required**: Yes  
**Format**: Numeric string (e.g., `"987654321"`)

Telegram chat ID of the admin user who receives approval notifications.

```toml
[vars]
ADMIN_CHAT_ID = "987654321"
```

**How to get**:
1. Message your bot
2. Run: `curl https://api.telegram.org/bot<BOT_TOKEN>/getUpdates`
3. Find `"chat":{"id":987654321,...}`

## Security Best Practices

<AccordionGroup>
  <Accordion title="Token Separation">
    Use **different tokens** for SDK and CLI:
    
    - `VAULT_ACCESS_TOKEN` - SDK on servers, can't reach admin routes
    - `VAULT_CLI_TOKEN` - CLI on dev machine, can reach admin routes
    
    This ensures compromised servers cannot rotate keys or modify the vault.
  </Accordion>
  
  <Accordion title="Never Commit Secrets">
    Add to `.gitignore`:
    ```
    .env
    .env.local
    .vault-permit
    ~/.suron/
    ```
    
    Only `.env.suron` (encrypted) is safe to commit.
  </Accordion>
  
  <Accordion title="Rotate Tokens Regularly">
    Rotate `VAULT_ACCESS_TOKEN` every 90 days:
    
    1. Generate new token: `openssl rand -hex 32`
    2. Update Worker secret: `wrangler secret put VAULT_ACCESS_TOKEN`
    3. Update all servers' `.env` files
    4. Restart apps to pick up new token
  </Accordion>
  
  <Accordion title="Permit Token Storage">
    By default, permit tokens are RAM-only. To persist:
    
    - Store in environment variable platform (Render, Railway, etc.)
    - Use encrypted secret storage
    - Never commit to git
    - Aware that permit grants 24h vault access
  </Accordion>
  
  <Accordion title="Deploy Key Protection">
    Convex deploy keys have full database access:
    
    - Store only as Worker secret
    - Never in code or logs
    - Rotate if compromised:
      1. Create new key in Convex dashboard
      2. Update Worker: `wrangler secret put CONVEX_DEPLOY_KEY`
      3. Delete old key
  </Accordion>
</AccordionGroup>

## Platform-Specific Configuration

### Render

Add environment variables in dashboard:

1. Go to your service > Environment
2. Add:
   ```
   VAULT_WORKER_URL=https://...
   VAULT_ACCESS_TOKEN=abc123...
   VAULT_APP_ID=my-app
   ```
3. Click **Save Changes** (triggers redeploy)

### Railway

Add via CLI or dashboard:

```bash
railway variables set VAULT_WORKER_URL=https://...
railway variables set VAULT_ACCESS_TOKEN=abc123...
railway variables set VAULT_APP_ID=my-app
```

### Fly.io

Set secrets:

```bash
flyctl secrets set VAULT_WORKER_URL=https://...
flyctl secrets set VAULT_ACCESS_TOKEN=abc123...
flyctl secrets set VAULT_APP_ID=my-app
```

### Vercel

Add in project settings:

1. Project Settings > Environment Variables
2. Add variables for Production, Preview, Development
3. Redeploy to apply

### Docker

Pass via `-e` flag:

```bash
docker run \
  -e VAULT_WORKER_URL=https://... \
  -e VAULT_ACCESS_TOKEN=abc123... \
  -e VAULT_APP_ID=my-app \
  my-app:latest
```

Or use `.env` file:

```bash
docker run --env-file .env my-app:latest
```

### Kubernetes

Create secret:

```yaml
apiVersion: v1
kind: Secret
metadata:
  name: suron-vault
type: Opaque
stringData:
  VAULT_WORKER_URL: https://...
  VAULT_ACCESS_TOKEN: abc123...
  VAULT_APP_ID: my-app
```

Reference in deployment:

```yaml
spec:
  containers:
  - name: app
    envFrom:
    - secretRef:
        name: suron-vault
```

## Validation Checklist

Before deploying, verify:

<Checklist>
  - [ ] Worker deployed with all 7 secrets set
  - [ ] `ADMIN_CHAT_ID` in `wrangler.toml`
  - [ ] Telegram webhook configured
  - [ ] Convex backend deployed
  - [ ] CLI logged in (`~/.suron/auth.json` exists)
  - [ ] Server has 3 env vars: `VAULT_WORKER_URL`, `VAULT_ACCESS_TOKEN`, `VAULT_APP_ID`
  - [ ] Tokens match between Worker secrets and server `.env`
  - [ ] App registered (`suron init` completed)
  - [ ] Vault encrypted and uploaded (`suron encrypt` completed)
</Checklist>
